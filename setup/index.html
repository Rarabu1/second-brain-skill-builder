The stack

Obsidian is the dashboard. Claude Code is the steering wheel.

This guide treats Obsidian as your editor and visual surface, and Claude Code — running inside Obsidian's integrated terminal — as the thing that reads, writes, and decides. Same vault folder, two interfaces on it. Cowork shows up later for the specific jobs it does better.

Install commands below match your pick. Change it anytime.

1. Obsidian

Free markdown editor. Your vault is just a folder of .md files — nothing proprietary. From Community Plugins, install Terminal (integrated terminal pane), Tasks (checkbox tracking), and optional Dataview (live queries over your files). Also install the Obsidian Web Clipper browser extension — it lets you save any article, page, or highlighted passage directly to your inbox while you're reading. Context that would otherwise vanish ends up in your vault instead.

A note on plugins: Obsidian community plugins run with full access to your vault and your computer, not in a sandbox. Terminal runs real shell commands and Templater can execute code, so only install plugins you trust, stick to well-known ones, and keep them updated. The four above are widely used and open source.

Windows note: the Terminal plugin's integrated pane needs Python 3.9+ installed, plus pip install psutil pywinctl typing_extensions. Set the Python path in the plugin's settings and click its "Check" button to confirm. (Macs already ship with a usable Python.)

2. Claude Code, installed and ready

Open the Terminal pane inside Obsidian (or Terminal.app) and run curl -fsSL https://claude.ai/install.sh | bash. The native installer is a standalone binary — no Node.js, no Xcode tools, no admin password. Then type claude to start. Before you paste it: this downloads a script from Anthropic and runs it on the spot, so only ever pipe a web address into your shell when you trust the source. If you use Homebrew, brew install --cask claude-code is a signed alternative that checks the download for you. Verify: claude --version prints a version number. If you get command not found, close and reopen the terminal so it picks up the new command, then retry.

Open PowerShell (not CMD) and run irm https://claude.ai/install.ps1 | iex. The native installer is a standalone binary — no Node.js, no WSL, no Administrator. Then type claude to start. Before you paste it: this downloads a script from Anthropic and runs it on the spot, so only run it when you trust the source. winget install Anthropic.ClaudeCode is a signed alternative that checks the download for you. Verify: claude --version prints a version number. If you see 'irm' is not recognized, you're in CMD — switch to PowerShell (its prompt starts with PS C:\).

Run it inside Obsidian's Terminal pane with your vault folder open and you're in a session that can read and write every file in the vault. Obsidian's file watcher picks up changes instantly — new files appear in the explorer as Claude writes them. Claude Code requires a paid Claude plan (Pro, Max, Team, or Enterprise) or an API key — the free Claude.ai plan doesn't include it. The first time you run claude it opens your browser to sign in.

3. Hosted connectors for the tools you live in

Claude Code now ships with hosted connectors for Gmail, Google Calendar, Google Drive, GitHub, and more — nothing to install. The first time a routine tries to use one, Claude prompts you in the terminal to authenticate via browser. Authorize once and it persists across sessions, which means a long-lived access token is stored on your machine (under ~/.claude), so treat that folder like a password and revoke the connector from the provider's account settings if the machine is ever lost or shared. Pick read-only scopes when the OAuth screen offers them. (If you need a service that doesn't have a hosted connector, you can self-host an MCP server with claude mcp add <name> -- <command>, but most people don't need this.)

4. Cowork, for two specific jobs (later)

The Claude desktop app's Cowork mode handles scheduled tasks (briefs that run while your laptop sleeps) and live artifacts (dashboards that re-query connectors on open) better than Claude Code does. You don't need Cowork on day one — it shows up in Chapter 5. See Appendix B for the trade-offs and when to switch.

Chapter 01

The Foundation

The problem

Every Claude session starts from zero. You explain your role, your projects, your terminology, how you like to communicate. Every. Single. Time. Then Claude invents a definition for one of your internal programs that's completely wrong, and you spend five minutes correcting it. Multiply that by three sessions a day.

What you build

A workspace with two files Claude reads automatically at the start of every session: CLAUDE.md (your identity, projects, preferences, rules) and soul.md (how Claude should behave). Plus a PARA folder structure — Projects, Areas, Resources, Archives — that gives every future file a clear home and keeps the vault from becoming a dumping ground.

Before

"Hi Claude, I'm a product marketing manager at a fintech. I lead positioning for our payments product. We have an initiative called LIFTOFF for new merchant onboarding and another called ANCHOR for retention — please don't conflate them. My manager is Priya. I'm working on three things right now…"

After

Claude opens the workspace, reads CLAUDE.md, and already knows all of this. You say "check the status of the Q3 positioning push" and Claude goes straight to the project file.

The PARA structure you'll end up with

workspace/ ├── CLAUDE.md your identity, projects, rules ├── soul.md personality config ├── 00_inbox/ capture everything new │ ├── meetings/ │ ├── notes/ includes follow_ups.md │ ├── triage/ daily triage files land here │ └── incubator/ half-baked ideas ├── 01_projects/ active work with a deadline ├── 02_areas/ ongoing responsibilities │ └── routines/ triage, brief, prep routines ├── 03_resources/ people, meetings, references │ ├── meetings/ │ └── people/ ├── 04_archives/ completed projects │ └── triage/ └── 05_reports/ generated outputs

The prompt

In Obsidian's terminal pane, with claude running in your vault folder, paste the prompt below. Claude will create the entire structure and you'll see the new folders and files appear in Obsidian's explorer as they're written.

prompt Set up the PARA workspace, CLAUDE.md, and soul.md 
I want to set up a PARA-based Second Brain workspace in this folder. Create:

The PARA directory structure:
- 00_inbox/ (subfolders: meetings/, notes/, triage/, incubator/)
- 01_projects/
- 02_areas/ (subfolder: routines/)
- 03_resources/ (subfolders: meetings/, people/)
- 04_archives/ (subfolder: triage/)
- 05_reports/

A follow-ups file at 00_inbox/notes/follow_ups.md (empty for now).

An incubator CLAUDE.md at 00_inbox/incubator/CLAUDE.md with lifecycle rules: capture (new idea folders land here), accumulate (add context over time), promote (move to 01_projects/ when ready — 3+ context additions or referenced 2+ sessions), archive (remove stale ideas).

A root CLAUDE.md with these sections (I'll fill in my details after):
- About Me (placeholder)
- Directory Structure (the PARA tree)
- File Naming Conventions (snake_case, zero-padded, no spaces, YYYY_MM_DD dates)
- Active Projects (empty table)
- Completed Projects (empty table)
- Personality (reads soul.md at session start)
- AI Agent Instructions (be direct, be proactive, no flattery, read project CLAUDE.md first)
- Communication Rules (never summarize what I said, verify before sending)
- Key Contacts (empty table)
- Domain Knowledge (placeholder)

A soul.md with this personality configuration:
- Core Identity: calm, precise, conversational. A collaborator, not a chatbot.
- Voice: measured and confident, no hedging, no filler.
- Pushback: challenge bad ideas directly. State what's wrong and offer the better path.
- Proactivity: surface improvements and missed opportunities without being asked.
- Drafting: write the full draft in my voice. Flag choice points, don't ask permission mid-draft.
- Uncertainty: say "I don't know" and ask whether to dig deeper.
- Hard Rules: no sycophancy, no over-explaining, no permission-seeking for obvious next steps, no fake empathy, no throat-clearing, no emojis.

After creating everything, remind me to fill in About Me, Key Contacts, and Domain Knowledge.

Verify it worked

Ask Claude to list the markdown files it created. You should see at minimum:

Files that should now exist

Now fill in your details

Open CLAUDE.md in Obsidian and fill in three sections. (You can also ask Claude Code to walk you through them — it'll prompt you section by section and write the answers in.)

About Me. Your name, role, team, location, and one paragraph about what you focus on. The more specific you are, the better Claude calibrates. "I lead a backend team of six at a healthtech startup, currently driving the migration from a monolith to event-driven services" is better than "I'm an engineer."

Key Contacts. Start with 3–5 people you interact with most. Include their role and any notes that matter ("prefers async over meetings", "weekly Wednesday sync", "communicates better in writing than on calls").

Domain Knowledge. This is the most important section for preventing hallucination. Define every term, program, or acronym your specific work uses that Claude's training data might get wrong. Be explicit about what things are not.

Pick the role closest to yours for an example:

Domain knowledge — policy / partnerships
  • BRIDGE: our stakeholder engagement program. NOT the inter-agency working group (that's the COUNCIL).
  • The framework: refers to the regulatory compliance framework adopted in Q1, not the older reporting framework.
  • ORCA: an industry coalition we participate in. We did NOT found it.
Domain knowledge — engineering
  • INGEST: our event pipeline service. NOT the analytics warehouse (that's PRISM).
  • The migration: refers to the monolith → event-driven services work, not the database migration we did last year.
Domain knowledge — marketing
  • LIFTOFF: new merchant onboarding campaigns. Distinct from ANCHOR (retention).
  • The repositioning: the brand work for the new audience segment, not the product naming exercise from Q1.
Domain knowledge — operations
  • Board pack: the quarterly deck for the Series A board. Distinct from the monthly investor update.
  • The all-hands: bi-weekly company meeting. Not the weekly leadership sync.

If you skip this section, Claude will confidently make up definitions for your domain terms. Every correction you have to make later could have been prevented here.

Tips

  • Initialize git for rollback safety. Git is version control — it takes a timestamped snapshot of your files on demand so you can undo anything Claude writes. Cloud sync is not backup; it just mirrors the current state. Have Claude run git init && git add -A && git commit -m "V1: foundation" in your vault, then commit after each chapter. You can roll back any of Claude's edits.
  • soul.md is worth the effort. Without it, Claude defaults to assistant-mode ("Great question! I'd be happy to help!"). With it, you get a collaborator that challenges bad ideas and does the obvious next thing without asking permission.
  • Use a SessionStart hook to auto-load soul.md. Claude Code can run a small script automatically when each session starts. One line in a settings file tells it to load soul.md without you having to remember. (See Appendix A for the exact snippet to paste in — it takes two minutes.)
  • Don't overthink the project list. Start with zero projects. You'll create your first one in Chapter 2.
Chapter 02

Memory + Your First Project

The problem

You corrected Claude last week: "Priya is the marketing lead, NOT the legal contact." Today, Claude calls Priya the legal contact again. You also have a project that spans weeks. Status changes, documents get created, reviews come back. But every session you reconstruct the context from scratch.

What you build

A memory system Claude uses automatically to persist corrections, preferences, colleague context, and behavioral feedback across sessions. Optionally, a human-readable corrections log for an audit trail. And your first structured project with a CLAUDE.md that tracks status, documents, and pending reviews.

Before

"No, I told you yesterday, Priya is marketing. Maria is legal. And the positioning doc is with Jane for review, I sent it Tuesday."

After

Claude remembers Priya is marketing (saved as a memory), never calls your post-incident program "real-time" again (saved as feedback), and knows the positioning doc went to Jane on Tuesday because the project CLAUDE.md tracks it.

How memory works

Claude Code has a built-in memory system that lives in ~/.claude/memory/ (or under your vault if you point it there). You don't have to set it up. When you correct Claude or share a preference, Claude saves a small typed memory file — user (who you are), feedback (how to work with you), project (what's happening), or reference (where to find things). An index file called MEMORY.md is loaded into every future session, so corrections never repeat.

You can see and edit your memories any time. Ask "show me my memories" and Claude will list them. If something is wrong, ask Claude to update or remove it. The files are plain markdown — you can open and edit them in Obsidian directly.

What memory looks like in practice

Stop summarizing what you just did at the end of every response.
Understood. I won't recap at the end of responses going forward.
saved: feedback memory
The program is called LIFEBOAT, not LIFEBELT.
Got it — LIFEBOAT. I'll update my memory and the corrections log.
saved: project memory (correction)
Jane moved to the legal team last month.
Updated — Jane is now on legal. I'll route review threads accordingly.
updated: colleague memory

Optional: human-readable corrections log

Some people want an audit trail they can scan in Obsidian. If that's you, run this prompt:

prompt Add a corrections log alongside native memory 
Create a corrections log at 03_resources/corrections.md with four sections:
- Names (table: corrected from, corrected to, date)
- Programs (same)
- Framing Rules (table: rule, context, date)
- Org Relationships (same)

When I correct you in the future, save a memory as usual AND append the correction here so I have a human-readable record.

This file is for you, not for Claude. Native memory does the behavioral work.

Your first project

Replace the bracketed items with your actual project. A few example first projects from different roles:

First project — policy

Federal Rulemaking Response — drafting comments on a proposed regulation affecting your industry. Phase: internal review.

First project — engineering

Auth Service Migration — moving from legacy session-based auth to a new token service. Phase: shadow traffic testing.

First project — marketing

First project — operations

prompt Create my first project 
Create my first project.

- Description: [e.g., "Brand work for the new audience segment, launching October"]
- Current phase: [e.g., "Creative review"]

Create it at 01_projects/01_[slug]/ with a CLAUDE.md containing:

- When sending a doc for review, add it to Pending Review.
- When review completes, update the status.

Tips

  • Corrections are the highest-value memories. In a domain with specific terminology, Claude's training data will confidently give you wrong definitions. Memory is the fix.
  • People get confused. If you have two colleagues with similar names, correct it the first time it happens — Claude will save them as distinct memory entries.
  • Don't duplicate between CLAUDE.md and memory. Project status goes in the project's CLAUDE.md. Cross-cutting preferences go in memory. If the same fact lives in both places, delete one.

Chapter 03

Morning Autopilot

The problem

You open your laptop and the chaos starts. Six chat threads, forty emails, a calendar with back-to-back meetings. An hour later you haven't done any real work. And you missed a direct ask from your manager buried in an unnamed group chat.

What you build

Three routines Claude follows step by step: chat triage reads your unread Slack/Teams conversations and classifies them by priority. Email triage does the same for Gmail. Morning brief orchestrates both plus calendar, follow-ups, project status, and tasks.

Before

An hour of scrolling, mentally sorting, sometimes missing things.

After

"Morning brief." Claude runs all three triage steps, scans your calendar, checks your follow-ups, and presents a consolidated brief. The five things that need your attention are at the top with suggested actions.

Before you start: The triage routines use Claude Code's hosted connectors for Gmail, Calendar, and your chat tool. You don't install anything — the first time a routine fires, Claude prompts you in the terminal to authenticate each connector via browser. Click through and authorize read access. Done in 60 seconds, once.
prompt 3A Chat triage routine 
Create a chat triage routine at 02_areas/routines/chat_triage.md:

# Chat Triage Routine
Triggered by: "triage my chat", "check my messages"

## Step 1: Find Unreads
Use the Slack connector (or whichever chat tool I have connected) to list all conversations with unread messages. Use last 3 days for first triage (catches weekends), last 4 hours for midday/EOD.

## Step 2: Read ALL Unreads
CRITICAL: Read EVERY conversation with unreads. Do not skip unnamed groups or low-profile DMs. Important coordination hides in small unnamed threads.

## Step 3: Classify
Action Required — ALL THREE must be true:
- I am the bottleneck (no one else can or will act)
- There is a concrete action I can take right now
- Delay creates real cost (missed deadline, blocked partner, escalation)

FYI — useful context, but I'm not blocking anyone.
Noise — skip entirely.

Common mistakes:
- "Someone else responded" doesn't downgrade if the action is still mine
- "CC'd" or "looped in" often means I'm expected to act
- Unnamed groups are NOT lower priority

## Step 4: Present
Action Required items first (cap at 5). For each:
- Bold the person/channel name
- 2–3 sentence summary
- Clickable link from the connector
- Suggested response in italics
- "Why" line: why I'm the bottleneck and what happens if I don't act

Then FYI items. Skip noise entirely.
Bottom Line: top 2–4 concrete actions.
If nothing needs attention: "All quiet."
prompt 3B Email triage routine 
Create an email triage routine at 02_areas/routines/email_triage.md:

# Email Triage Routine
Triggered by: "triage my email", "check my email"

## Step 1: List Unreads
Use the Gmail connector to list unread messages. Skip newsletters, automated notifications, digest emails, and calendar invites.

## Step 2: Read Important Emails
Read the full body of emails from known contacts, my manager chain, or external partners.

## Step 3: Classify
Same framework as chat triage. Additional strong signals:
- From my manager chain: likely Action Required
- External partner expecting a reply: likely Action Required
- Older than 48h and I haven't replied: flag it

## Step 4: Present
Same format as chat triage. Include a clickable link for every item.

Bottom Line: N actionable, N FYI, N skipped. Flag unanswered emails older than 48h.
prompt 3C Morning brief — the orchestrator 
Create the morning brief at 02_areas/routines/morning_brief.md:

# Morning Brief
Triggered by: "morning brief", "catch me up", "what did I miss"

## Execution Order
Run steps 1–6 sequentially:

### Step 1: Chat Triage
Execute 02_areas/routines/chat_triage.md

### Step 2: Email Triage
Execute 02_areas/routines/email_triage.md

### Step 3: Calendar
Use the Google Calendar connector to pull today's events.
- Today's meetings with attendees and agenda
- Flag meetings in the next 2 hours needing prep
- Note external partner meetings (higher priority)

### Step 4: Follow-ups
Read 00_inbox/notes/follow_ups.md. Surface overdue items.

### Step 5: Project Pulse
For each active project in root CLAUDE.md:
- Check the project's CLAUDE.md
- Flag pending reviews stale >3 days
- Flag upcoming deadlines
- Only surface items where I am the bottleneck. Cap at 5.

### Step 6: Tasks
- Open tasks I own
- Tasks I created (assigned to others)
- Status changes or stale items

## Output
Write to 00_inbox/triage/YYYY_MM_DD.md as checkboxes:

### Action Required (max 5, from all sources)
- [ ] Source — what's needed. [link]
      Why: bottleneck + action + cost of delay
      Suggested: specific action

### Chat (FYI only)
- [ ] Person — summary. [link]

### Email (FYI only)
- [ ] Sender — summary. [link]

### Calendar
Time | Meeting | Flags

### Follow-ups and Project Pulse
One line per item.

## Variants
- Midday ("midday check"): chat last 4h + afternoon calendar only.
- EOD ("wrap up"): chat last 4h + tomorrow calendar + session notes.

## Cross-Session Continuity
- Read today's triage file if it exists (skip handled items).
- Read yesterday's file for unresolved items.
- Run fresh scan, dedupe, append new items only.
- Items marked done are never re-surfaced.

Then update root CLAUDE.md Session-Start Routine: "At session start, offer to run the morning brief."

Why the three-test framework matters

The biggest risk with any triage system is over-promotion: everything looks urgent when you read it fresh. The three-part test (bottleneck + concrete action + real cost of delay) forces genuine prioritization.

A real example: a partner sends a message saying "following up on last week's meeting." Your instinct says Action Required. But are you the bottleneck? Did someone else already respond? Is there a specific deadline? Without the test, you mark everything red and end up with 15 "urgent" items instead of 5.

Tips

  • Read ALL unreads. The most important item is often in an unnamed group chat you'd normally skip. Don't let the format of the channel determine the priority.
  • Verify before classifying. Did you already handle this? Check the thread, check the linked doc, check yesterday's triage. Re-promoting stale items is the most common triage mistake.
  • The triage file is your persistent to-do list. Writing it as Obsidian checkboxes means you can check items off as you handle them, and Obsidian's Tasks plugin will roll them up into a vault-wide dashboard.
  • Promote routines to skills once they stabilize. A routine markdown file works by name reference ("execute morning_brief.md"). Move the file into .claude/skills/morning-brief/SKILL.md and Claude Code autoloads it with richer triggers. The skill-creator skill walks you through it.
Don't move on until the routines actually fire when you trigger them.
Chapter 04

Meeting Prep

The problem

You walk into a meeting with a partner and realize you don't remember what was discussed last time, what commitments your team made, or what the partner's current position is on a contentious question. You spend the first ten minutes catching up, and the partner notices.

What you build

Two tiers of meeting prep. Quick prep (under 5 minutes) pulls context from your people files, project files, and recent triage. For routine syncs. Full prep pipeline dispatches four research agents in parallel to scan docs, chat, email, and your project history simultaneously. For high-stakes meetings with decisions at stake.

Before a partner meeting

"Let me dig through email for the last exchange… check three chat threads… find the strategy doc… remind myself who else will be in the room…" (twenty minutes.)

After

"Full prep for the partner sync." Claude reads the invite, looks up attendees, scans relevant threads via the connectors, reads the strategy doc, checks the project file for outstanding commitments, and produces talking points calibrated to your actual position. Five minutes of your time while agents run in parallel.

Why people files matter more than you think

An AI assistant kept confusing two colleagues with similar first names. One worked on partnership operations, the other on external communications. Drafts were going to the wrong person. People files fixed it permanently: each person has a distinct identity, role, and communication style.

People files also compound. After a few months, Claude knows that your CFO prefers three-bullet summaries, your legal counsel wants everything in writing, and your main external partner responds faster on chat than email. These are the kinds of things that make Claude's output feel uncannily well-calibrated.

You'll feel the payoff the first time you walk into a meeting fully prepped in five minutes.
Chapter 05

Automation

The problem

Up to here, Claude only runs when you have a session open. If you want a brief waiting when you wake up, someone has to start the session and run the prompt. That someone is you. At 8 AM. Before coffee.

What you build

Three things, in order of effort. Scheduled briefs via launchd so the morning brief writes itself before you wake up. A live "Today" view built with Obsidian's Dataview plugin, or a static HTML report Claude regenerates on demand. PARA maintenance: a routine that runs as part of the morning brief and keeps the workspace from accumulating cruft.

Heads up: This is the chapter where Cowork starts pulling ahead for two specific jobs — scheduled tasks on Anthropic's infrastructure (no laptop required) and live dashboards that re-query connectors. If the launchd approach below feels like overkill, jump to Appendix B for the Cowork path. Both work against the same vault.

Scheduled briefs with launchd (Mac)

launchd is macOS's built-in job scheduler — it can run a script automatically at any time you set, even while you're asleep or away from the computer. The idea here: launchd wakes at 7:55 AM, starts claude in non-interactive mode, hands it the morning-brief prompt, restricts it to a minimal set of tools, and lets it write the triage file. Your laptop has to be awake — if you sleep with it closed, Cowork's scheduled tasks handle this without needing the machine on.

prompt 5A Generate the launchd plist for me

How the interactive session uses the scheduled output

At the start of each interactive triage, Claude should: read the scheduled task's latest brief, extract observations as raw data, discard the scheduled task's priority classifications (stale snapshots without full context), run a fresh scan, merge with the observations, verify each item, and apply independent priority classification using full context.

The interactive session's value is judgment, not reformatting. The scheduled run provides raw signal. You provide the context that makes it meaningful.

Safety boundaries

Pre-send verification. Before Claude sends ANY outbound message (email, chat, etc.), it should present a verification block and wait for your explicit "go":

VERIFY: Channel=[name], To=[recipients], Tone=[formal/casual/external]
[draft message]
Reply "send" to confirm.

Add this rule to root CLAUDE.md Communication Rules. Autonomous action boundaries: Claude reads files, sorts inbox, updates people files, flags items, writes triage files without asking. Claude always asks first before sending messages, archiving projects, deleting files, or closing follow-ups.

Least-privilege connectors. When the OAuth screen asks for scopes during authentication, pick read-only wherever offered. Triage and scheduled briefs only need to read. The --allowedTools flag in your launchd plist narrows what Claude can reach to the tools you list, so keep that list minimal and leave Bash out: a single Bash tool can run any command, which defeats a read-only setup. Treat this as a strong default, not an unbreakable wall.

Watch what the daemon reads, not just what it sends. A scheduled run ingests untrusted outside content: emails, calendar invites, messages. Someone who can send you any of those can plant instructions in that content, hoping an unattended agent acts on them. This is why the daemon stays read-only and Bash-free, and why you should read its triage output before acting on it. The pre-send verification above protects outbound messages; this protects against malicious inbound content.

A note on thresholds. The graduation signals (4 of 5) and incubator promotion criteria (3+ context additions) are reasonable defaults, not researched thresholds. Adjust based on experience. If projects linger after meeting the criteria, lower the bar. If ideas get promoted too early, raise it.

Which model to use

  • Haiku for scheduled runs. Fast and cheap. Use it for the launchd daemon — it reads files, runs the brief, writes the triage output. You don't need Sonnet's reasoning for that job.
  • Sonnet for daily interactive work. Triage, drafting, meeting prep, most prompts. The right default for sessions where you're doing real work.
  • Opus for high-stakes output. Complex analysis, anything where judgment matters more than speed, drafts that go to leadership. Slower and more expensive — worth it when it counts.

SessionStart hook — auto-load soul.md and corrections

Drop this into <vault>/.claude/settings.json (create the folder and file if they don't exist). Claude Code will run the cat commands at the start of every session in this directory, loading your personality config and corrections log into context automatically.

{
  "hooks": {
    "SessionStart": [
      {
        "type": "command",
        "command": "cat soul.md"
      },
      {
        "type": "command",
        "command": "test -f 03_resources/corrections.md && cat 03_resources/corrections.md || true"
      }
    ]
  }
}

The test -f … || true on the second hook lets it run even before you've created the corrections log in Chapter 2.

Appendix B

When Cowork is the better tool.

The Claude desktop app's Cowork mode is the right answer for a few specific jobs. The vault stays the same — both Claude Code and Cowork can mount the same folder and use the same routine files. Pick per job, not per philosophy.

Five places Cowork pulls ahead

Scheduled briefs while your laptop sleeps

Cowork's scheduled tasks run on Anthropic's infrastructure. An 8 AM morning brief fires whether your machine is awake or not. launchd needs your laptop on and online. If you close your laptop at night, this is the single best reason to keep Cowork around.

A live "Today" dashboard you check five times a day

Cowork's artifacts re-query connectors every time you open the page. Claude Code can write static HTML or use Dataview against local files, but neither refreshes connector data on its own. If you'd reload a dashboard repeatedly to see what's new, Cowork's the win.

Sharing the system with non-technical colleagues

Most people won't install Node, configure MCPs, or learn launchd. Cowork's connectors panel is a one-click OAuth flow. If you want a partner or a less-technical friend running the same Second Brain pattern, point them at Cowork — same prompts, same chapters, no terminal.

Connector quality for Google Drive, Notion, Asana

Cowork has first-party connectors that just work. MCP servers exist for most of these, but quality varies and some require self-hosting. The first-party path saves a weekend of yak-shaving on the ones where MCPs are flaky.

Cross-device access

Cowork lives in the desktop app and increasingly on the web. Claude Code is terminal-only. If you sometimes triage from an iPad or check the day's brief on your phone, Cowork covers that.

How to use both without conflict

Point Cowork's folder mount at your Obsidian vault. Both Claude Code and Cowork now read and write the same files. Use Claude Code in Obsidian's terminal for the daily steering wheel (drafting, triage interactively, project moves) and Cowork for the cron-style jobs (scheduled briefs, live artifacts, mobile check-ins). The routine markdown files don't care which is reading them.

Migrating a single job to Cowork (without leaving Claude Code)

The scheduled morning brief. Disable the launchd plist (launchctl unload ~/Library/LaunchAgents/com.user.secondbrain.morningbrief.plist). In Cowork, create a scheduled task with the same prompt body. The output path stays the same — Cowork writes to 00_inbox/triage/YYYY_MM_DD.md in your vault, Obsidian renders it, your interactive Claude Code session reads it the same way.

The "Today" view. Keep the Dataview note for the static cross-vault rollup. Add a Cowork artifact for the live connector data (calendar, unread Action Required, @-mentions). They cover different jobs — Dataview for your own files, the artifact for live external state.

Quick Start

If you read this far and just want to start.

  1. Install Obsidian. Open (or create) your vault. Install the Terminal, Tasks, and (optional) Dataview community plugins. On Windows, the Terminal plugin also needs Python 3.9+ and pip install psutil pywinctl typing_extensions.
  2. Install Claude Code (macOS): curl -fsSL https://claude.ai/install.sh | bash (Windows, in PowerShell): irm https://claude.ai/install.ps1 | iex. Requires a paid Claude plan (Pro/Max/Team/Enterprise); the free plan doesn't include it.
  3. Open the Terminal pane inside Obsidian. Confirm you're in your vault folder, then type claude to start your first session.
  4. Nothing to install for connectors — Gmail, Calendar, and chat are hosted. The first time a routine uses one, Claude prompts you to authenticate via browser. Pick read-only scopes when asked.
  5. Paste the Chapter 1 prompt. Fill in About Me, Key Contacts, and Domain Knowledge.
  6. Start using it.

Add chapters as you feel the need. Bring Cowork in when you hit Chapter 5 if you want scheduled briefs while your laptop sleeps. The system grows with you.
